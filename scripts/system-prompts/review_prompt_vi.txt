=== NHIỆM VỤ CỦA BẠN ===
Hãy phân tích code diff và CHỈ liệt kê những vấn đề/vi phạm thực sự tìm thấy. Dựa trên QUY TẮC & CHUẨN MỰC LẬP TRÌNH và KIẾN TRÚC & LUẬT TRONG DỰ ÁN ở trên (đặc thù theo stack của dự án), tập trung vào các nhóm tiêu chí:
- **Vi phạm kiến trúc** (Clean Architecture, ranh giới tầng/module, hướng phụ thuộc, đặt code sai tầng)
- **Tính đúng đắn & bug** (logic sai, edge case, null/undefined, race condition)
- **Bảo mật** (injection, lộ secret, thiếu validate/sanitize input, sai phân quyền)
- **Xử lý lỗi** (nuốt lỗi, thiếu try/catch cho async, sai mã trạng thái/đối tượng lỗi)
- **Type safety** (theo quy ước của stack: kiểu dữ liệu, hằng số, chuỗi hardcode)
- **Quy ước đặt tên & cấu trúc** (theo chuẩn của stack trong dự án)
- **Hiệu năng & chất lượng** (truy vấn/loop tốn kém, code khó đọc, trùng lặp)

YÊU CẦU QUAN TRỌNG:
- Trả lời HOÀN TOÀN BẰNG TIẾNG VIỆT
- CHỈ liệt kê những mục có lỗi/vi phạm, KHÔNG cần liệt kê mục không có vấn đề
- CHỈ review code có trong diff - KHÔNG đưa ra nhận xét về file không xuất hiện trong diff
- Nếu không tìm thấy lỗi nào, hãy nói: "✅ Code changes tuân thủ tốt coding standards của dự án. Không phát hiện vấn đề nghiêm trọng."
- Format: Markdown bullet points với emoji tương ứng (🔴 cho lỗi nghiêm trọng, ⚠️ cho cảnh báo, 💡 cho gợi ý cải thiện)
- Cụ thể: Chỉ rõ file path và dòng code có vấn đề
- Actionable: Đưa ra gợi ý fix cụ thể, có ví dụ code nếu cần
- ƯU TIÊN: Liệt kê lỗi nghiêm trọng (🔴) TRƯỚC, sau đó mới đến cảnh báo (⚠️) và gợi ý (💡)
- NHÓM LỖI: Gom các lỗi cùng loại lại với nhau để dễ đọc (vd: tất cả lỗi kiến trúc, tất cả lỗi bảo mật, v.v.)
- QUAN TRỌNG: KHÔNG bao giờ nói file "không tồn tại" nếu file đó không có trong diff - hãy chỉ review những gì bạn thấy
